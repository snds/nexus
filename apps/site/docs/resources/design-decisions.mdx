# Design decision records

Significant decisions are captured as DDRs so the *why* survives the decision. Each carries a
stable ID for cross-tool reference.

## DS-2026-001 — Canvas background follows the theme token

- **Status:** Active · **Severity:** Critical (a11y) · **Date:** 2026-06-05
- **Context:** The React Flow canvas pane was pinned to near-black, so in **light** theme the graph
  sat on a dark plane against light chrome — a 1.22:1 context mismatch and a jarring read.
- **Decision:** Bind the canvas background to `--nx-bg` (the theme-flipping semantic) instead of a
  fixed color.
- **Rationale:** The token already flips correctly per theme; the canvas should obey the same
  cascade as every other surface. Light-mode contrast became **16.8:1**.
- **Tradeoffs:** None material — the dark theme is visually unchanged.

## DS-2026-002 — Generic primitives compose Radix; only graph pieces are bespoke

- **Status:** Active · **Severity:** High · **Date:** 2026-06-07
- **Context:** Hand-rolled overlays/menus risked inconsistent focus, escape, and ARIA.
- **Decision:** Build Button, Tooltip, Dialog, DropdownMenu, and Popover on **shadcn/Radix** bases
  themed through the token bridge; wrap them in thin product primitives. Only genuinely
  domain-specific graph pieces (NodeBadge, GraphLegend) are bespoke.
- **Rationale:** Real focus management, portals, and keyboard nav for free; less surface to test
  and maintain.
- **Tradeoffs:** A Radix/shadcn dependency — accepted for the accessibility and maintenance win.

## DS-2026-003 — Tokenize text & a primitive color layer

- **Status:** Active · **Severity:** High · **Date:** 2026-06-07
- **Context:** The system had font-family only — no type scale, weights, or primitive color ramps —
  so type and color weren't truly tokenized or documented.
- **Decision:** Add L0 primitive ramps (neutral + accent) and a full type scale, with a `Text`
  component exposing semantic variants. Document **all** of it, not just values in use.
- **Rationale:** A design system documents the whole range; consumers need the scale, not only the
  subset the product happens to render today.

## DS-2026-004 — Re-base color on Radix Colors + the 12-step contract

- **Status:** Active · **Severity:** High (a11y + architecture) · **Date:** 2026-06-08
- **Context:** A `/qa` audit measured three WCAG AA contrast failures — primary button (white on
  accent, **2.83:1**), destructive (3.91), and light-mode subtle text (3.82). Root cause was
  architectural: the hand-rolled HSL ramps violated the Radix step contract — text was drawn from
  **step 9** (a *solid* step, not a text step), and accent was used for both fills and text.
- **Decision:** Adopt `@radix-ui/colors` as L0 primitives — **Slate** (neutral) + **Indigo**
  (accent) + **Red** (destructive). Primitives are single-mode flat constants (light + dark side by
  side); modes live only at L1 semantics. Remap every semantic to its correct Radix step: text →
  **11/12**, solids → **9/10**, borders → **6–8** (8 = ring). Split `--nx-accent` (solid, step 9)
  from `--nx-accent-text` (step 11). Consume as `var()`; opacity via Tailwind utilities /
  `color-mix`, not inline hsl alpha.
- **Why Indigo over Blue:** white on Radix Blue-9 is only **3.26:1** (Radix step-9 targets APCA/3:1,
  not WCAG-small). White on **Indigo-9 = 5.21:1** — a solid, on-brand primary that passes AA.
- **Result (measured, both themes):** primary **5.21**, fg-subtle **5.79–9.06**, accent-text
  **5.85–9.15** — all AA. Locked by a CI contrast test.
- **Tradeoffs accepted:** (1) **Destructive** solid (Red-9 + white) is **3.91** — Radix reds can't
  carry white at AA (same physics as Blue); kept as the standard solid destructive (passes AA-large
  / APCA / 3:1), the single documented exception. Strict-AA alternative = the "soft" recipe
  (red-3 bg + red-11 text). (2) Three text tiers collapse to two AA-safe steps (11/12); finer
  hierarchy is by weight/size, not a sub-AA color. (3) Entity/severity remain single-tone primitives
  (full Radix-scale mapping is a tracked fast-follow).
