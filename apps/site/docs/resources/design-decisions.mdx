# Design decision records

Significant decisions are captured as DDRs so the *why* survives the decision. Each carries a
stable ID for cross-tool reference.

## DS-2026-001 — Canvas background follows the theme token

- **Status:** Active · **Severity:** Critical (a11y) · **Date:** 2026-06-05
- **Context:** The React Flow canvas pane was pinned to near-black, so in **light** theme the graph
  sat on a dark plane against light chrome — a 1.22:1 context mismatch and a jarring read.
- **Decision:** Bind the canvas background to `--nx-bg` (the theme-flipping semantic) instead of a
  fixed color.
- **Rationale:** The token already flips correctly per theme; the canvas should obey the same
  cascade as every other surface. Light-mode contrast became **16.8:1**.
- **Tradeoffs:** None material — the dark theme is visually unchanged.

## DS-2026-002 — Generic primitives compose Radix; only graph pieces are bespoke

- **Status:** Active · **Severity:** High · **Date:** 2026-06-07
- **Context:** Hand-rolled overlays/menus risked inconsistent focus, escape, and ARIA.
- **Decision:** Build Button, Tooltip, Dialog, DropdownMenu, and Popover on **shadcn/Radix** bases
  themed through the token bridge; wrap them in thin product primitives. Only genuinely
  domain-specific graph pieces (NodeBadge, GraphLegend) are bespoke.
- **Rationale:** Real focus management, portals, and keyboard nav for free; less surface to test
  and maintain.
- **Tradeoffs:** A Radix/shadcn dependency — accepted for the accessibility and maintenance win.

## DS-2026-003 — Tokenize text & a primitive color layer

- **Status:** Active · **Severity:** High · **Date:** 2026-06-07
- **Context:** The system had font-family only — no type scale, weights, or primitive color ramps —
  so type and color weren't truly tokenized or documented.
- **Decision:** Add L0 primitive ramps (neutral + accent) and a full type scale, with a `Text`
  component exposing semantic variants. Document **all** of it, not just values in use.
- **Rationale:** A design system documents the whole range; consumers need the scale, not only the
  subset the product happens to render today.

## DS-2026-004 — Re-base color on Radix Colors + the 12-step contract

- **Status:** Active · **Severity:** High (a11y + architecture) · **Date:** 2026-06-08
- **Context:** A `/qa` audit measured three WCAG AA contrast failures — primary button (white on
  accent, **2.83:1**), destructive (3.91), and light-mode subtle text (3.82). Root cause was
  architectural: the hand-rolled HSL ramps violated the Radix step contract — text was drawn from
  **step 9** (a *solid* step, not a text step), and accent was used for both fills and text.
- **Decision:** Adopt `@radix-ui/colors` as L0 primitives — **Slate** (neutral) + **Indigo**
  (accent) + **Red** (destructive). Primitives are single-mode flat constants (light + dark side by
  side); modes live only at L1 semantics. Remap every semantic to its correct Radix step: text →
  **11/12**, solids → **9/10**, borders → **6–8** (8 = ring). Split `--nx-accent` (solid, step 9)
  from `--nx-accent-text` (step 11). Consume as `var()`; opacity via Tailwind utilities /
  `color-mix`, not inline hsl alpha.
- **Why Indigo over Blue:** white on Radix Blue-9 is only **3.26:1** (Radix step-9 targets APCA/3:1,
  not WCAG-small). White on **Indigo-9 = 5.21:1** — a solid, on-brand primary that passes AA.
- **Result (measured, both themes):** primary **5.21**, fg-subtle **5.79–9.06**, accent-text
  **5.85–9.15** — all AA. Locked by a CI contrast test.
- **Tradeoffs accepted:** (1) **Destructive** solid (Red-9 + white) is **3.91** — Radix reds can't
  carry white at AA (same physics as Blue); kept as the standard solid destructive (passes AA-large
  / APCA / 3:1), the single documented exception. Strict-AA alternative = the "soft" recipe
  (red-3 bg + red-11 text). (2) Three text tiers collapse to two AA-safe steps (11/12); finer
  hierarchy is by weight/size, not a sub-AA color. (3) Entity/severity remain single-tone primitives
  (full Radix-scale mapping is a tracked fast-follow — done in DS-2026-005).

## DS-2026-005 — Full Radix palette + channels map to hue stacks

- **Status:** Active · **Severity:** Medium · **Date:** 2026-06-08
- **Context:** Entity/severity/context were single-tone hex values, so a channel had a fill color but
  no usable bg/border/text steps; and only the in-use hues existed.
- **Decision:** Ship **all 31 Radix scales** (25 colors + 6 grays, light + dark) as flat L0
  primitives (`--nx-<scale>-<light|dark>-<1…12>`, in `radix-scales.css`). Map every entity (14),
  severity (6), and context/alert (4) channel onto a **Radix hue stack** (documented assignment);
  each channel's token resolves to its hue's solid step 9, with the full stack reachable for
  bg/border/text variants. Resolves the DS-2026-004 fast-follow.
- **Rationale:** Any hue stack is now readily usable across the UI, and channels inherit Radix's
  per-step contrast guarantees instead of hand-tuned one-offs (Radix step 9 is consistent enough
  across light/dark to drop the old manual light-mode darkening).
- **Tradeoffs:** ~770 lines of generated primitives (isolated in `radix-scales.css`); the palette
  is intentionally complete even though the product uses a subset — a design system documents the
  whole range.

## DS-2026-006 — Color-as-identifier strategy: reserved zones + shared intent axis

- **Status:** Active · **Severity:** High · **Date:** 2026-06-08
- **Context:** Channels were assigned hues ad hoc, conflating two encoding jobs. Hue is a
  **nominal-only** channel (Bertin; Cleveland & McGill), yet verdict severity (ordinal) was a hue
  rainbow, and entity *type* (nominal) borrowed the semantic danger hues (`actor→red`,
  `malware→crimson`) — so a node's color falsely signaled danger and risked red/green CVD confusion.
- **Decision:** Split color-as-identifier into two systems, disambiguated by **encoding role +
  form** (revised from the initial disjoint-zone idea — see Amendment):
  - **Intent / outcome** (ordinal + semantic): one shared `--intent-*` ramp — danger (red),
    caution (orange), warning (amber), success (grass), info (blue), neutral (gray). **Verdict
    severity and status alerts both alias it.** Verdict is a **diverging** ramp (worst→best) read by
    the warm→cool progression, not a rainbow. It always appears as a small pip/badge with icon+label.
  - **Identity** (nominal): entity *type* → **8 super-category families**, each a maximally-distinct
    hue (≥39° apart in OKLCH, full wheel): adversary=crimson · artifact=plum · identifier=orange ·
    file=yellow · host=teal · web=cyan · comms=iris · meta=gray. Specific type within a family is
    read by **shape + icon**.
- **Rationale:** Matches the encoding-theory contract (`uid-color-for-ui`, `infod-encoding-theory`):
  hue=nominal → identity; ordinal severity → diverging ramp; categorical color is reliable only to
  ~8 hues, so 14 types collapse to 8 family hues + redundant shape/icon. Identity and intent share
  the wheel but never confuse because they differ in **form** (large node fill vs small pip/badge)
  and both pair color with shape/icon/label.
- **Amendment (form-based, supersedes disjoint-zone):** The first cut reserved the entire warm arc
  for intent, confining identity to the cool/purple region — which crowded 14 hues into ~⅓ of the
  wheel (measured min gap **4°**, far below the ~30° at-a-glance threshold). Relaxing to form-based
  separation reopened the full wheel; the 8-family palette now sits at **≥39° min gap** with
  lightness variation. Green stays out of identity (reserved as the benign/safe signal) and pure
  danger-red stays out as a node fill.
- **Tradeoffs:** (1) Within a family two types share a hue (e.g. actor & campaign = crimson), relying
  on shape + icon — the correct response to the ~8-color limit, and arguably richer (color = family,
  icon = type). (2) Identity and intent hues can coincide (e.g. identifier-orange vs caution-orange),
  accepted because form disambiguates. (3) `phishing` sits in the danger band (crimson) rather than
  as a unique verdict, preserving the ramp's monotonic order.

## DS-2026-007 — One source for the node language (legend ≡ canvas)

- **Status:** Active · **Severity:** Medium · **Date:** 2026-06-08
- **Context:** The node *shape* came from one taxonomy (`shapeOf` — 4 shape groups) while *color*
  came from another (the 8 color families), so the canvas and the legend grouped the same nodes
  two different ways. They didn't match.
- **Decision:** Make the **family the single source** for both channels. `ENTITY_FAMILIES`
  (`@nexus/domain`) now defines each family's `hue` **and** `shape`; `ENTITY_META[type].family`
  joins a type to it; `shapeOfType` / `familyOf` derive everything. `nodeShape.shapeOf` delegates to
  the domain source, the canvas/minimap render shape + color from the family, and the **GraphLegend
  family header is a mini-node** — the family's shape silhouette in its color. Legend and canvas are
  identical by construction.
- **Model:** shape = 4 macro-classes (circle threats · square infrastructure · diamond
  detections/identifiers · hexagon comms) that the 8 color families **nest into** — so shape + color
  reinforce one identity (a colorblind-safe redundancy), and the icon reads the specific type.
- **Tradeoffs:** Some types changed shape to align with their family (e.g. exploit diamond→circle as
  an artifact). Two families can share a shape (distinguished by color); within a shape, color +
  icon disambiguate.
