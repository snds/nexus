## Two color systems, two rule sets

Color-as-identifier splits into two orthogonal systems drawn from **disjoint hue zones**, so they
never cross-talk (see [Foundations → Color](/docs/foundations/color)):

- **Intent / outcome** (ordinal + semantic) — verdict severity and status alerts both alias one
  shared ramp. Reserved hues: red · orange · amber · grass · blue · gray. Always with an icon + label.
- **Identity** (nominal) — entity *type*, drawn only from the **non-reserved** hues (purples,
  teals, metals…), grouped into families, paired with shape + icon. A type color **never** uses a
  reserved intent hue, so a node's color can't falsely signal danger.

Each channel maps to a full Radix **hue stack**: the token resolves to step 9, and bg (step 3),
border (6), and text (11) of that hue are reachable via `--nx-<hue>-<light|dark>-<step>`.

## Entity (color = type) — 8 super-category families

Categorical color is reliable to ~8 hues, so **color encodes 8 super-category families** (each a
maximally-distinct hue, ≥39° apart in OKLCH across the full wheel); the **specific type within a
family is read by shape + icon**. Intent/verdict may share these hues — it's kept distinct by
*form* (a small pip/badge with icon + label), not by an exclusive zone. Green is intentionally
absent (reserved as the "benign/safe" signal) and pure danger-red stays out of identity.

| Family | Hue | Types |
|---|---|---|
| Adversary | crimson | actor · campaign |
| Malicious artifact | plum | malware · exploit |
| Identifier | orange | hash · sid |
| File | yellow | filename |
| Host | teal | ip · hostname |
| Web | cyan | url · domain |
| Comms | iris | email · message |
